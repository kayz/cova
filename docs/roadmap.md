# COVA Roadmap

## 目标

在保证架构可扩展的前提下，先完成一个可联调、可异步交付、可追溯证据的专家系统最小闭环。

## 里程碑总览（10 周）

| Milestone | 周期 | 目标 | 核心交付 | 验收标准 |
| --- | --- | --- | --- | --- |
| M0 基线搭建 | Week 1 | 建立工程骨架 | 目录结构、CI、基础规范、文档导航 | PR 可自动检查，文档可追踪 |
| M1 契约冻结 | Week 2 | 定义稳定外部接口 | API 契约、错误模型、Job 状态机、Webhook 事件模型 | `coco` 可按契约联调 |
| M2 异步内核 | Week 3-4 | 跑通任务闭环 | Gateway、Orchestrator、Queue、状态持久化 | `submit -> status -> result` 全流程可用 |
| M3 固收样例专家 | Week 5-6 | 验证专家接入模式 | RSS ingestion、KB、daily brief worker | 每日综述可生成且可回传 |
| M4 稳定性与安全 | Week 7-8 | 提升可靠性与安全性 | 幂等、重试、超时、DLQ、签名回调、审计日志 | 故障可恢复，敏感字段受控 |
| M5 平台化准备 | Week 9-10 | 面向多专家扩展 | capabilities、路由策略、Go SDK、第二个 mock 专家 | 新专家可在不改主接口下接入 |

## 当前执行重点（从这里开始）

当前建议从 **M1 契约冻结** 开始，而不是先做内容能力优化。

原因：

1. 主 Agent 只应该依赖稳定接口，不依赖专家实现细节。
2. 异步任务一旦上线，状态机和事件模型后改成本很高。
3. 契约先稳，后续专家扩展才能复用。

## 近两周执行计划

## Sprint A（Week 1）

- [ ] 冻结最小接口集合：
  - [ ] `POST /v1/jobs/brief`
  - [ ] `GET /v1/jobs/{job_id}`
  - [ ] `GET /v1/jobs/{job_id}/result`
  - [ ] `POST /v1/query`
- [ ] 定义 Job 状态机与转换规则。
- [ ] 定义 webhook 事件格式与签名规则。
- [ ] 完成 `docs/api-contract.md` 并评审通过。

验收：

- `coco` 侧可基于文档开发 client stub。
- 所有参与方对字段语义无歧义。

## Sprint B（Week 2）

- [ ] 实现 mock 异步 worker（先不接 RSS/LLM）。
- [ ] 实现最小任务持久化与状态查询。
- [ ] 实现 webhook 回调重试（指数退避）。
- [ ] 产出联调脚本（提交任务、查状态、取结果）。

验收：

- 10 次连续调用中，任务闭环成功率 100%（mock 场景）。
- 回调失败可重试并可观测。

## 每个里程碑的 Definition of Done

每个里程碑完成需同时满足：

1. 接口文档更新完毕并与实现一致。
2. 至少 1 个端到端联调用例可重复执行。
3. 关键指标可观测（延迟、成功率、错误分布）。
4. 回滚方案明确（功能开关或版本回退）。

## 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 接口反复变更 | 联调延期 | M1 冻结契约，变更走版本策略 |
| 外部模型不稳定 | 任务失败率升高 | 超时 + 重试 + 降级兜底 |
| RSS 质量波动 | 综述质量不稳定 | 数据源评分、去重、质量过滤 |
| 回调不可靠 | 调用方收不到结果 | webhook + 轮询双通道 |

## 优先级规则

执行中出现冲突时按以下优先级取舍：

1. 契约稳定性 > 新功能数量
2. 可观测性 > 快速上线
3. 异步可靠性 > 单次任务时延
4. 可追溯证据 > 生成文本长度
